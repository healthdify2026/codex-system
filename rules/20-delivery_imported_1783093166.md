# 20 交付标准与版本管理规则

> 优先级低于 00-security.md 与 10-global.md。

## 版本管理与文件覆盖（关键约束）

- **deliverables/ 与 reviews/ 内的定稿类文件永不覆盖**：任何更新、整理、审核、修订，一律按小版本 +0.1 另存为新文件（如 v1.0 → v1.1）；若使用者明确指定版本号，按指定执行。
- 文件名格式：`名称_v版本_日期`，如 `数据治理方案_v1.2_20260703`。
- work/ 与 research/ 内的过程稿允许原地修改，回滚与追溯由 Git 承担，不叠加手工版本号。
- input/ 内原始资料只读；需要加工时复制到 work/ 再动。

## 通用交付标准

- 交付物一律存项目 deliverables/ 目录。
- 定稿四项硬标准：目标回应完整、全文术语一致、无占位符与待补残留、关键结论来源可追溯。
- L3 及以上任务的交付物，定稿前必须完成双 AI 交叉验证（Claude 出方案 → ChatGPT 独立挑错 → Fable 综合裁决 → 人拍板），裁决记入 DECISIONS.md，完整意见存入 reviews/。
- 面向客户的交付物不出现内部工具名、模型名与内部流程细节。

## 文档类

- 结构齐全：背景与目标 → 方案主体 → 落地步骤 → 风险与边界。
- 医疗、数据类方案必须包含「数据安全与合规」独立章节。

## 代码类

- Codex 产出需附自述说明：做了什么、改了哪些文件、如何验证。
- 经 Opus 依 task-brief 验收标准审查通过后方可定稿；写的不审自己。
- 迭代整改最多 2 轮，超过转人工；每轮审查意见追加至 task-brief 尾部留痕。
- 不交付无法本地复现的代码；依赖与运行方式写入 README 或自述说明。

## 内容创作类

- 稿件版本以文件为准（v0 草稿 / v1 修订 / vF 定稿），不以对话记录为准。
- 发布前过 40-domains/content.md 的发布检查单（该文件遇到首个内容项目时再建）。
