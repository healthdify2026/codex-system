# 10 全局规则

> 优先级低于 00-security.md，高于项目层规则。所有 Claude / Codex 会话开始工作前应读取本文件。

## 身份与语言

- 使用者为独立顾问，常驻日本，工作领域：AI 项目、医疗信息化、数据管理与治理、数据安全、大健康/医养/康养、小说编剧与短视频创作。
- 默认使用简体中文交流与产出；代码与代码注释可用英文。

## 工作方式

- 文件是资产，对话是草稿：有价值的结论必须当场落盘到 codex-system 内对应文件，未落盘视为未发生。
- 一处主写：每个项目任一时刻只有一台主机在写（规划/评审期为 MacBook，实施期为 Mac mini）；移动端产生的内容一律进 inbox/，不直接改项目文件。
- 项目目录六分区：input（只读原始资料）、research（调研分析）、work（过程稿）、reviews（审核意见）、deliverables（定稿）、以及项目卡与任务决策文件。
- 所有产出先落 work/；定稿须经人确认后才移入 deliverables/。
- 修改任何已有文件前，先说明改动范围与理由；删除与覆盖必须人工确认。

## 任务分派

- 任务按 L1–L4 分级（低/中/高/极高风险），等级由 Fable 建议、人确认；拿不准就高不就低。
- L2 及以上任务派发前必须写 task-brief（八字段：目标、输入、输出、禁改范围、验收标准、最大迭代次数、失败处理、人工确认点）。
- Claude 模型金字塔：Fable＝规划拆解收口与重大裁决；Opus＝深度推理、方案主笔、审查 Codex 产出；Sonnet＝日常主力（默认）；Haiku＝批量轻活。
- 审查原则：写的不审自己，审的不改文件。

## 记录习惯

- 决策写入项目 DECISIONS.md：append-only，格式为「日期 + 决策 + 理由」。
- 审核与交叉验证意见存入项目 reviews/。
- 任务状态实时更新 TASKS.md（待办 / 进行 / 待审 / 完成 / 待人工）。
- 对话中出现值得长期保留的约定，当场追加进 rules/ 对应文件并 Git 提交。
